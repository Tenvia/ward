# Ward v0.1.0-rc2 — real egress evaluation plan

## Status

Planning document only. No RC2 core feature implementation starts until this plan is reviewed and explicitly approved.

## RC2 thesis

RC1 proved the installable demo and control-plane containment story: a tenant can be constrained or paused while another tenant keeps running, with approval evidence and audit evidence. RC2 should prove the next, narrower thing: Ward can credibly sit in front of realistic staging AI traffic without hiding its prototype boundaries.

The RC2 theme is **Real Egress Evaluation**. The release target is not a broader gateway, not an observability product, and not a production deployment story. It is a Phase 1 design-partner proof slice: streaming responses, verified upstream pass-through, observe-only adoption, a unit-test floor, and honest docs/claims.

## What RC2 must prove

1. Ward can handle `stream: true` requests on `POST /v1/chat/completions` without weakening tenant enforcement.
2. Enforcement decisions happen before the first streamed upstream byte.
3. A configured upstream pass-through path is exercised by repeatable checks before any stronger claim ships.
4. A no-cost mock/fixture path exists so the verification battery does not depend on paid provider calls.
5. Observe-only mode can run with zero tenant-state enforcement blocking risk for attributed proxied traffic while emitting visible would-block or would-constrain evidence; validation, auth, and configured fail-closed errors still apply.
6. The core containment primitives have a small automated unit-test baseline.
7. Published claims stay at `planned`, `prototype`, or `demo-supported` until the matching checks pass and the claims ledger is updated.

## Must-build items

1. **Unit test baseline**
   - Tenant state transitions.
   - Detection threshold behavior.
   - Approval-token single-use, TTL, and exact-phrase rules.
   - Audit append/list behavior.
   - Observe-only would-block evidence after observe-only exists.

2. **Observe-only mode**
   - Instance default configuration: `observe` / `enforce`, or an equivalent small mode model, with an explicit per-tenant override path for mixed rollout.
   - In observe-only mode, paused or constrained tenant state must never block otherwise-valid proxied traffic.
   - The would-have-blocked decision must be visible through audit evidence; response headers and structured JSON logs may be supplemental but cannot by themselves upgrade the claim.
   - Response headers may expose an explicit would-block marker for caller visibility, but only as supplemental to the required audit event.

3. **Verified upstream pass-through mode**
   - Exercise the existing real-upstream path before claiming it.
   - Add a gated live smoke that runs only when upstream environment variables are present.
   - Add a no-cost local mock upstream path for default verification.

4. **Streaming support for chat completions**
   - Support `stream: true` for `POST /v1/chat/completions`.
   - Keep missing tenant header as `400`.
   - Keep paused tenant as `423` in enforce mode.
   - Keep constrained tenant as `429` in enforce mode.
   - In observe-only mode, pass the request while recording would-block evidence.
   - Stream running-tenant responses from the configured upstream or the mock streaming fixture.

5. **Latency benchmark**
   - Measure proxy overhead instead of asserting it.
   - Cover mock and pass-through paths where safe.
   - Publish numbers only with the command, environment, and caveats.

6. **Docs, claims, and release notes sync**
   - Update docs only after evidence exists.
   - Add claims ledger rows or update existing rows only when checks support the exact status.
   - Keep RC2 notes explicit about remaining prototype boundaries.

## Should-build items

1. Minimal response-header evidence for observe-only decisions, if it does not create API ambiguity.
2. A static SSE fixture for deterministic streaming checks.
3. A small local mock upstream helper for pass-through and streaming smokes.
4. Latency output that is easy to paste into release notes without hand-editing numbers.
5. Control Room observe-only surfacing only if it stays trivial and does not delay the data-plane proof.

## Out-of-scope items

1. Request Shaping / Cost Containment.
2. Postgres, Redis, queues, or a new durable event system.
3. Production auth, RBAC, OIDC, per-operator identity, or token rotation.
4. Kubernetes runner work or NetworkPolicy enforcement.
5. Hosted Ward.
6. Generic AI gateway behavior, model routing, payload transformation, token optimization, or provider policy management.
7. Complete provider API coverage beyond the request and response shapes exercised by RC2 checks.
8. Production-grade audit retention, export, compliance posture, HA, or hard-down survival.
9. Commits, tags, pushes, publishes, or release workflows without explicit owner approval.

## Acceptance criteria

### Unit test baseline

- A project-standard unit-test command exists and is documented.
- The command covers tenant state transitions, detection thresholds, approvals, audit append/list behavior, and observe-only would-block behavior.
- Tests are deterministic and do not require Docker, a browser, paid provider calls, or a live external service.

### Observe-only mode

- Enforce mode keeps the current blocking semantics.
- Observe-only mode passes otherwise-valid proxied traffic for tenants that would be paused or constrained under enforce mode.
- Mixed-mode rollout is covered by an explicit per-tenant override: an observe-default instance can enforce for one configured tenant while another tenant remains observe-only.
- Observe-only mode emits visible would-block or would-constrain audit evidence; headers/logs are supplemental only.
- The evidence includes tenant ID, attempted action/path, effective mode, state that would have blocked, and timestamp or existing audit metadata.
- No doc claims production-grade durable audit.

### Pass-through mode

- Default verification uses a local mock upstream or fixture.
- A live upstream smoke is gated behind explicit environment variables and skips loudly when they are absent.
- The pass-through path is not described as broadly provider-compatible beyond what the smoke covers.

### Streaming

- `stream: true` works for a running tenant through the mock streaming path.
- Enforce-mode paused/constrained tenants are blocked before any upstream stream is opened.
- Observe-only paused/constrained tenants are allowed through while recording would-block evidence.
- Missing tenant header stays a pre-upstream `400`.
- The implementation does not buffer the entire upstream stream before returning data.

### Latency benchmark

- A repeatable command reports proxy overhead for the supported RC2 paths.
- The command prints environment and sample-size context.
- Release notes include numbers only after a successful run.

### Docs and claims

- `docs/CLAIMS_AND_EVIDENCE.md` reflects only evidenced statuses.
- `docs/BUILD_STATUS.md` records commands and results.
- Release notes label RC2 capabilities as prototype/demo-supported according to evidence.
- Known gaps remain visible: single-node prototype, shared-token auth, SQLite prototype, hard-down gap, and limited provider compatibility.

## Verification commands

These are planned gates; exact scripts may change during implementation, but RC2 should not ship without equivalent evidence.

```bash
npm run validate:openapi
npm run test:unit
npm run smoke:demo
npm run smoke:reliability
npm run smoke:openapi
npm run smoke:pass-through:mock
npm run smoke:streaming:mock
npm run smoke:observe-only
npm run bench:latency
./scripts/verify-release.sh
```

Optional live upstream gate, skipped unless explicitly configured. Do not paste real secrets into shell history; set them through the local env manager. The required gate is equivalent to:

```bash
WARD_UPSTREAM_OPENAI_BASE_URL=<configured> OPENAI_API_KEY=<configured> npm run smoke:pass-through:live
```

Before any release action, also verify the user Docker path and Control Room browser checks remain green:

```bash
./scripts/smoke-user-install.sh
cd apps/control-room && npm run test:e2e && npm run test:e2e:auth
```

## Claims rules

- Use the exact status vocabulary from `docs/CLAIMS_AND_EVIDENCE.md`.
- If a claim has no ledger row, it is not claimed.
- `tested` requires automated unit or integration coverage for the specific behavior.
- `demo-supported` requires a smoke, E2E, or scripted demonstration that exercises the behavior.
- Do not claim broad provider API coverage, SLA/uptime, production deployment posture, production RBAC, production-grade audit, Postgres support, hosted Ward, K8s runner behavior, or Request Shaping / Cost Containment.
- Prefer phrasing such as `planned`, `target`, `verification gate`, `prototype`, and `demo-supported` until evidence justifies an upgrade.

## Release blockers

1. Streaming enforcement opens an upstream stream before deciding tenant state.
2. Observe-only mode blocks traffic in any paused/constrained case.
3. Would-block evidence is missing, invisible, or stored in a new unreviewed durable subsystem.
4. Pass-through checks depend only on paid live calls and lack a default no-cost path.
5. Unit tests are absent for the core state/approval/detection/audit seams.
6. Latency numbers are asserted without a repeatable command.
7. OpenAPI changes are undocumented or unvalidated.
8. Docs imply stronger support than the verification gates prove.
9. Saastle, hosted Ward, Postgres, RBAC/OIDC, K8s, or gateway-scope work enters RC2.
10. Any tag, publish, push, or release workflow is attempted without explicit approval.


## Implementation architecture and order

This order is deliberate: first create cheap proof seams, then add the adoption mode, then prove real egress, then add streaming, then measure and sync claims.

### 1. Unit test baseline

- **Intended files touched:** `apps/api/package.json`, root `package.json`, new API unit test files under `apps/api/src/` or `apps/api/test/`, and only small source seams if the current module-level state needs reset helpers.
- **Implementation sketch:** use Node's built-in test runner with the existing `tsx` devDependency rather than adding Jest/Vitest. Prefer an API-local command shaped like `node --import tsx --test \"src/**/*.test.ts\"`, then expose the root gate as `npm run test:unit`.
- **Test strategy:** behavior tests for `tenantState.ts`, `detection.ts`, `approvals.ts`, and `audit.ts`; keep them deterministic by using low env thresholds, short approval TTLs, and explicit reset helpers.
- **OpenAPI impact:** none.
- **Verification command:** `npm run test:unit`.
- **Risks:** current modules keep process-level state and read `config` at import time, so tests must isolate state without relying on import-order tricks.
- **Do not claim yet:** broad unit coverage, integration coverage, or `tested` status for endpoint behavior outside the covered seams.

### 2. Observe-only mode

- **Intended files touched:** `apps/api/src/config.ts`, `apps/api/src/types.ts`, `apps/api/src/server.ts`, `apps/api/src/audit.ts` if a helper is useful, `openapi/ward.v0.yaml`, `scripts/smoke-reliability.mjs` or a new observe-only smoke, `scripts/verify-release.sh`, docs/claims after evidence.
- **Implementation sketch:** add a small mode model such as `WARD_MODE=observe|enforce` with `enforce` as the default, plus a minimal per-tenant override mechanism for mixed rollout. In the existing `/v1/chat/completions` enforcement block, replace the direct `423`/`429` returns with a single pre-upstream decision helper: in enforce mode it returns the existing WardError response; in observe mode it logs `would_block`, sets a header such as `x-ward-would-block: paused|constrained`, and continues to `recordRequest` and upstream/mock forwarding.
- **Test strategy:** constrain and pause tenants under observe mode and assert `200` plus would-block header plus audit event; run the same states under enforce mode and assert the existing `429`/`423`; include one mixed-mode assertion proving one tenant enforces while another only observes.
- **OpenAPI impact:** add the observe mode health field or extend an existing health field, document `x-ward-would-block`, document the mode override surface if exposed over HTTP, and include the `would_block` audit action as prototype evidence. Keep `WardError` semantics unchanged for enforce mode.
- **Verification command:** `npm run smoke:observe-only` once added, plus `npm run validate:openapi`.
- **Risks:** observe-only must not accidentally override missing-header `400`, control auth `401`, or proxy fail-closed `503` unless the plan explicitly says so. The first slice should limit observe-only to tenant-state enforcement decisions.
- **Do not claim yet:** partner-safe rollout until the implementation and smoke prove those paths.

### 3. Pass-through verification

- **Intended files touched:** `apps/api/src/openaiProxy.ts`, `apps/api/src/config.ts` if a mock upstream base URL flag is needed, new or extended smoke script under `scripts/`, `package.json`, `scripts/verify-release.sh`, `docs/CLAIMS_AND_EVIDENCE.md` and `docs/BUILD_STATUS.md` after evidence.
- **Implementation sketch:** keep the existing real-upstream path minimal. Add a local mock upstream server inside the smoke script so default verification can exercise the `forwardToUpstream` branch without paid calls. Add a separate live smoke that runs only when `WARD_UPSTREAM_OPENAI_BASE_URL` and `OPENAI_API_KEY` are explicitly set and otherwise prints a loud skip.
- **Test strategy:** mock-upstream smoke asserts the API forwards the body, attaches authorization only to the mock upstream process, returns the upstream JSON/status where expected, and maps upstream failures to the existing `ward_upstream_error` path. Live smoke records only coarse status and redacts secrets.
- **OpenAPI impact:** likely none for the first verification slice unless response headers or error semantics change. Keep the existing `502` upstream-error contract.
- **Verification command:** `npm run smoke:pass-through:mock`; optional `WARD_UPSTREAM_OPENAI_BASE_URL=<configured> OPENAI_API_KEY=<configured> npm run smoke:pass-through:live`.
- **Risks:** the current `forwardToUpstream` assumes JSON response bodies; upstream SSE or invalid JSON will need separate streaming handling rather than being hidden in this smoke.
- **Do not claim yet:** broad provider compatibility, tool-call pass-through guarantees, or exhaustive OpenAI API parity.

### 4. Streaming path for chat completions

- **Intended files touched:** `apps/api/src/openaiProxy.ts`, `apps/api/src/server.ts`, `openapi/ward.v0.yaml`, `scripts/smoke-openapi-conformance.mjs`, a new streaming smoke under `scripts/`, `scripts/verify-release.sh`, docs/claims after evidence.
- **Implementation sketch:** branch on `req.body?.stream === true` only after tenant header validation, policy lookup, observe/enforce decision, request counting, and pressure detection. For enforce-mode paused/constrained tenants, return the existing non-stream WardError before opening any upstream connection. For observe mode, write would-block evidence and continue. For running tenants, pipe upstream SSE chunks through the Express response as they arrive; mock mode should emit a deterministic static SSE sequence. Do not buffer the complete stream.
- **Test strategy:** assert running tenant receives SSE chunks; missing tenant still gets `400`; paused/constrained enforce mode gets non-stream `423`/`429`; observe mode gets streamed data plus would-block evidence. Include a mock static streaming fixture so CI does not require a provider.
- **OpenAPI impact:** document `stream` on `ChatCompletionRequest`, add a `text/event-stream` response entry for `200`, keep JSON WardError responses for `400`/`423`/`429`/`502`/`503`, and keep honesty notes narrow.
- **Verification command:** `npm run smoke:streaming:mock`, `npm run smoke:openapi`, `npm run validate:openapi`.
- **Risks:** the race to avoid is sending SSE headers before enforcement completes. The route must complete all blocking decisions before `res.writeHead` or first `res.write`.
- **Do not claim yet:** complete streaming compatibility, provider-specific streaming edge cases, usage accounting from terminal chunks, or mid-stream recovery semantics beyond the tested behavior.

### 5. Latency benchmark

- **Intended files touched:** new `scripts/bench-latency.mjs`, root `package.json`, docs only after measured output exists.
- **Implementation sketch:** run repeated local calls against mock mode and the mock pass-through path, report p50/p95 and environment context. Keep live-provider benchmarking opt-in.
- **Test strategy:** benchmark script should fail on non-2xx mock-path responses and print sample count, warmup count, URL, mode, p50, and p95.
- **OpenAPI impact:** none.
- **Verification command:** `npm run bench:latency`.
- **Risks:** local laptop measurements are noisy; release notes must label the environment and avoid universal performance claims.
- **Do not claim yet:** latency SLA, production overhead, or provider-wide numbers.

### 6. Docs and claims sync

- **Intended files touched:** `docs/CLAIMS_AND_EVIDENCE.md`, `docs/BUILD_STATUS.md`, `docs/OBJECTIONS.md`, `docs/DEPLOYMENT_MODEL.md`, `docs/ARCHITECTURE.md`, `README.md`, `docs/releases/v0.1.0-rc2.md`, and OpenAPI honesty notes where behavior changed.
- **Implementation sketch:** update docs only after implementation checks pass. Add or revise claims ledger rows one atom at a time: streaming, pass-through checks, observe-only, unit baseline, and latency numbers each need separate evidence. Every upgraded public sentence must map to a claims-ledger row and a `docs/BUILD_STATUS.md` command result; otherwise it remains `planned`.
- **Test strategy:** use targeted stale-claim greps plus `npm run validate:openapi`; rerun only behavior checks tied to the claims being upgraded.
- **OpenAPI impact:** sync any changed endpoint behavior before prose claims.
- **Verification command:** `npm run validate:openapi` plus the exact smokes that back each upgraded row.
- **Risks:** the easiest failure is wording that sounds stronger than the ledger. Keep `planned` until the command output exists.
- **Do not claim yet:** any upgraded status without the exact command output in `docs/BUILD_STATUS.md`; defer broad exclusions to `Out-of-scope items` and `Claims rules`.

## Streaming race-condition plan

The streaming route must be structured as two phases:

1. **Pre-stream decision phase:** parse tenant header, compute operation/correlation ID, run policy lookup/fail-mode handling, evaluate tenant state, apply observe/enforce behavior, record request, detect pressure, and log any evidence. This phase may return JSON errors and has not sent SSE headers.
2. **Stream phase:** only after the pre-stream phase succeeds, open the upstream stream or mock stream, set `text/event-stream`, and pipe chunks without whole-response buffering.

The release-blocking assertion is: a constrained or paused tenant in enforce mode with `stream: true` receives a normal WardError response (`429` or `423`) and the mock/live upstream receives no request.

## Observe-only storage and evidence plan

RC2 should use existing prototype evidence channels only:

1. `logAudit()` with action `would_block` and evidence fields for `operation`, `correlationId`, `mode`, and `wouldHaveState`. This audit event is required for the observe-only claim.
2. SQLite-backed audit automatically when `WARD_STORAGE=sqlite`, because `audit.ts` already writes through the storage abstraction.
3. Optional structured stdout only as supplemental evidence if audit logging itself fails and the failure should not block observe traffic.
4. Optional `x-ward-would-block` response header for immediate caller-visible supplemental evidence.

Do not add Postgres, Redis, queues, a new event bus, or compliance-retention language in RC2.

## Pass-through and mock-upstream plan

Default verification should create a local in-process mock upstream inside the smoke script, point Ward at it through `WARD_UPSTREAM_OPENAI_BASE_URL`, and assert the `forwardToUpstream` branch was exercised. Live provider verification should be a separate opt-in command gated by explicit env vars and skipped loudly otherwise. Streaming tests should use a static SSE mock fixture first; no complex provider simulator is needed.

## Pause before implementation

Stop here for human review before core implementation begins. The approved plan should precede any source-code changes for streaming, observe-only mode, pass-through verification, unit tests, or latency benchmarking.
