# Ward v0.1.0-rc3 — RC3 release

This release covers the **Durable Operator Trust** thesis: when
Ward makes, blocks, or would-blocks a tenant containment decision,
an operator can trust the evidence after restart, review it later,
and explain exactly what happened.

## RC3 implementation summary (added 2026-07-05)

The RC3 plan was implemented across seven sessions ending on
2026-07-05. No published tag, image, or registry push was
performed during any RC3 slice. The plan sections that follow
this summary remain accurate as the Slices 0–8 plan layout; this
section records what shipped.

### What shipped (Slice 1 — Durable audit event model)

- `schemaVersion: 1` added to every emitted audit event
  (`apps/api/src/types.ts` literal type; `apps/api/src/audit.ts`
  sets it on every emission). The property is documented in
  `openapi/ward.v0.yaml` as `AuditEvent.schemaVersion: integer
  enum: [1]`.
- Two new unit tests pin the contract: every event carries
  `schemaVersion: 1`, and five consecutive events produce five
  distinct `audit_N` IDs.

### What shipped (Slice 2 — Audit restart recovery)

- No production change beyond the existing SQLite storage layer
  (`apps/api/src/storage/sqliteStore.ts`) — events already
  rehydrated on boot.
- `scripts/smoke-audit-durability.mjs` (root
  `smoke:audit-durability`): spawns Ward with `WARD_STORAGE=sqlite`,
  records a would-block event, kills and restarts the process with
  the same `WARD_SQLITE_PATH`, asserts the same event id is still
  present (44/44 checks, exit 0). Covers the new `schemaVersion`
  field round-trip on disk.

### What shipped (Slice 3 — Per-tenant mode override)

- `apps/api/src/types.ts` extended with `WardModeOverride =
  "inherit" | "observe" | "enforce"`. `TenantRecord` gains
  `modeOverride` (default `inherit`; legacy rows backfilled at
  load).
- `apps/api/src/effectiveMode.ts` (new): pure helper resolving
  the effective mode from the global setting plus tenant
  override. Four-case unit suite covers the matrix.
- `apps/api/src/server.ts`: `/v1/chat/completions` consults the
  helper before the paused/constrained block; the `would_block`
  audit event now carries `effectiveMode` and `override` in
  evidence.
- `apps/api/src/tenants.ts`: `POST /ward/tenants/:tenantId/mode`
  (control-auth-gated) with body shape
  `{ mode, actor?, reason? }`. Invalid values return 400
  `ward_invalid_mode_override`; override changes audit
  `mode_override_changed` with
  `evidence.{previousOverride, nextOverride}`.
- OpenAPI gains `WardModeOverride` enum, `TenantModeRequest`
  schema, the new path, and `modeOverride` on `Tenant`.
- `scripts/smoke-tenant-mode-override.mjs` (root
  `smoke:tenant-mode-override`): 38/38 covering override observe
  under global enforce, override enforce under global observe,
  inherit behavior, audit chain, and control auth.

### What shipped (Slice 4 — State transition contract)

- `apps/api/src/stateTransitions.ts` (new): pure helpers
  `isValidWardState`, `canTransitionTenantState`,
  `explainRejection`. Allow-table:
  `running → constrained`, `running → paused`,
  `constrained → running`, `constrained → paused`,
  `paused → running`; rejections for `paused → constrained` and
  any same-state transition.
- `apps/api/src/tenants.ts`: `applyTransition` consults the
  contract before `setState`; rejections return HTTP 409 with
  `{ error: "ward_invalid_transition", message, rejection,
  from, to }`.
- OpenAPI documents the rejection envelope as a
  `TransitionRejected` response on the four transition paths and
  a `TransitionRejection` schema.
- Unit tests: 14 in `stateTransitions.test.ts` (allow-table,
  same-state, unsupported, type guard); 4 in
  `tenants.transitions.test.ts` (wire-through).

### What shipped (Slice 5 — Incident receipt export)

- `scripts/export-incident-receipt.mjs` (root `incident:export`):
  reads Ward's SQLite store via Node's built-in `node:sqlite` or
  a live `/ward/audit`, emits a Markdown receipt with all
  required sections and a `## Limitations` block that disclaims
  compliance / tamper-proof / forensic framing. `--json` for
  machine output; `--out <path>` for file redirection.
- `scripts/smoke-incident-receipt.mjs` (root
  `smoke:incident-receipt`): 20/20 covering scenario +
  Markdown content + disclaimers.

### What shipped (Slice 6 — Failure-behavior smoke)

- `scripts/smoke-rc3-failure-behavior.mjs` (root
  `smoke:rc3-failure-behavior`): 37/37 covering audit restart,
  missing-tenant JSON, upstream failure (enforce + observe
  don't hide 502), enforce pre-upstream blocking (zero upstream
  hits while constrained), approval reuse (single-use +
  wrong-tenant rejection), stream + constrained returns 429
  JSON, and the `paused → constrained` 409 envelope shape.

### What shipped (Slice 7 — Docs + claims sync; this slice)

- `docs/CLAIMS_AND_EVIDENCE.md` — seven new RC3 prototype rows,
  refreshed `Live OpenAPI response conformance` note, claim
  rewrite rules extended for "durable audit trail" and
  "tamper-proof receipt" language.
- `docs/BUILD_STATUS.md` — RC3 session block at the top with the
  full verification table.
- `docs/releases/v0.1.0-rc3.md` — this implementation summary.
- `ROADMAP.md` — Phase 2 "Containment receipts" line moves from
  `planned` to `prototype, demo-supported`. Phase 3+ untouched.
- `docs/ARCHITECTURE.md` — "Out of scope today" refreshed to

## RC3 thesis

RC2 proved that Ward can credibly sit in front of realistic staging
AI traffic without hiding its prototype boundaries: streaming,
pass-through, observe-only, a unit-test floor, and a latency
benchmark were exercised against the existing local data plane.

RC3 should prove the next, narrower thing: **Durable Operator
Trust**. When Ward makes, blocks, or would-blocks a tenant
containment decision, an operator can trust the evidence after
restart, review it later, and explain exactly what happened.

RC3 builds on RC2, it does not replace it. The RC2 prototype
boundaries (single-node, shared-token auth, SQLite prototype,
hard-down gap, limited provider compatibility, no pass-through
streaming) remain in force unless an RC3 slice explicitly removes
one of them with matching evidence.

## What RC3 must prove

1. Ward's audit events are versioned and structured so an operator
   can read them later without relying on log scraping or
   reconstructive tools.
2. Audit evidence survives a process restart in the existing local
   persistent storage mode and the same evidence surfaces again on
   re-read.
3. An operator can export a human-readable incident receipt from the
   audit evidence: tenant, what happened, mode in effect at the
   time, what was blocked or would-blocked, what state changes
   occurred, who or what initiated them, what approvals were used,
   and what remains unknown or unverified.
4. Per-tenant observe/enforce overrides work alongside the global
   mode so one tenant can be observed while another is enforced,
   with override changes themselves audited.
5. The tenant state machine has an explicit, testable transition
   contract with previousState/nextState captured in audit and
   invalid transitions rejected with a clear WardError.
6. The RC3 trust-relevant failure behaviors have smoke coverage:
   persistent audit survives restart, missing tenant remains a JSON
   error, upstream failure remains clear, observe mode does not
   hide upstream failures, enforce mode still blocks before any
   upstream call, approval expiry/reuse still fails, blocked
   streaming responses remain JSON, misconfigured persistent
   storage surfaces a clear startup or health signal.
7. Public docs and the claims ledger reflect only the exact
   evidence Slice 7 produces; no status upgrades without matching
   command output in `docs/BUILD_STATUS.md`.

## What RC3 does NOT prove

- Production-grade durable audit, retention policy, or export to a
  compliance sink. The audit is local SQLite prototype with a
  human-readable Markdown export.
- Tamper-evident or signed receipts. The incident receipt is
  operational evidence, not legal or compliance certification.
- Hosted Ward, multi-node HA, queue middleware, or shared durable
  state across replicas.
- Postgres production backend.
- Production auth, per-operator RBAC, OIDC, token rotation.
- Real Kubernetes runner or K8s NetworkPolicy enforcement.
- Production-grade request shaping, real dollar-cost metering,
  budget-based constrained mode.
- Complete OpenAI compatibility, broad provider support, or
  pass-through streaming against a real upstream.
- A published `v0.1.0-rc3` tag, pushed branch, or GHCR release. No
  RC3 slice performs any tag, push, or publish action without
  explicit owner approval.

## Target slices

Slice 0 — RC3 planning alignment (this document). Read listed
files; produce the RC3 plan; no code.

Slice 1 — Durable audit event model. Add a `schemaVersion`, stable
event ID, timestamp, tenant ID, action, actor, reason,
previousState/nextState, evidence payload, and correlationId where
useful. Preserve existing audit behavior and smoke/test
compatibility unless intentionally updated.

Slice 2 — Durable audit persistence and restart recovery. Confirm
audit events rehydrate from the existing local persistent storage
backend after a process restart; add a smoke
(`scripts/smoke-audit-durability.mjs`, root script
`smoke:audit-durability`) that starts Ward, triggers a would-block
event, stops it, restarts it with the same storage, and confirms
the event is still present.

Slice 3 — Per-tenant mode override. Global `WARD_MODE` keeps
defaulting enforce/observe; tenants can set `inherit`, `observe`,
or `enforce`; effective mode is computed per tenant; override
changes are audited; existing global behavior remains the
default; no "off" mode and no silent bypass.

Slice 4 — State transition contract. Define allowed transitions
between running, constrained, and paused. Require reason for
operator-driven state changes where the API supports it. Preserve
previousState/nextState in audit. Reject invalid transitions with
a clear WardError. Preserve approval expiry/reuse behavior.

Slice 5 — Incident receipt export. Add
`scripts/export-incident-receipt.mjs` (root script
`incident:export` or `export:incident`) producing a Markdown
receipt that answers who, what, mode, blocked or would-blocked
content, state changes, actor, approvals, supporting evidence,
and limitations. Optional JSON if simple. Add
`scripts/smoke-incident-receipt.mjs` (`smoke:incident-receipt`)
that generates a small scenario, exports, and verifies required
fields.

Slice 6 — Failure-behavior verification. Add
`scripts/smoke-rc3-failure-behavior.mjs`
(`smoke:rc3-failure-behavior`) covering the high-value trust
cases: persistence restart, missing-tenant JSON error, upstream
failure clarity, observe-mode honesty, enforce-mode pre-upstream
blocking, approval expiry/reuse, blocked-streaming JSON, and
persistent-storage misconfiguration signals.

Slice 7 — Docs and claims sync. Update only with verified
evidence: README, ROADMAP, CLAIMS_AND_EVIDENCE, BUILD_STATUS,
ARCHITECTURE, DEPLOYMENT_MODEL, and this release doc.

Slice 8 — Final RC3 verification. Run the full RC3 battery from
this plan plus the RC2 verify-release harness if it composes
cleanly. Produce the final evidence report.

## Claims allowed in RC3 (only after Slice 7 evidence exists)

- Ward has versioned local audit events.
- Ward can persist audit evidence across restart in local
  persistent mode.
- Ward supports per-tenant observe/enforce mode override.
- Ward has an explicit tenant state transition contract.
- Ward can export a local incident receipt from audit evidence.
- Ward has RC3 failure-behavior smoke coverage.

## Known risks inherited from RC2

These gaps exist today and remain open unless an RC3 slice
explicitly closes one with matching evidence. Slice 0 lists them
so reviewers see the starting surface; Slice 7's claims sync
must not let any of them leak into a claim without evidence.

- Streaming pass-through against a real upstream is untested.
- Mid-stream cancel is not audited.
- Live-provider latency is not measured.
- Hard-down proxy fallback is unsolved; if the Ward process is
  fully down, Ward cannot pass traffic.
- Single-node only; no multi-replica failover.
- Shared-token auth only; no rotation, roles, or per-operator
  identity.
- Storage is the SQLite prototype; no retention policy or
  export-to-audit-sink.
- Hybrid proxy/governance reach is narrower than a gateway;
  this is by design, but broad upstream coverage is not
  claimed.

## Implementation architecture and order

The slice order is deliberate. Each slice ends in a clean,
verified state before the next begins, and each independent
slice stays small enough that the trust claim it adds can be
reviewed on its own.

### Slice 1 — Audit event model

- Intended files touched: `apps/api/src/types.ts`,
  `apps/api/src/audit.ts`, unit tests under
  `apps/api/src/*.test.ts`, possibly
  `apps/api/src/storage/types.ts`.
- Implementation sketch: introduce a `schemaVersion` field on the
  audit event with a small default; keep existing event shape
  backward-compatible; tag the storage backend to persist the
  new field through both memory and SQLite paths so a later
  Slice 7 reader can rely on it.
- Tests: shape, unique IDs, schemaVersion present, evidence
  preserved, list ordering, reset behavior.
- OpenAPI impact: none in Slice 1.
- Verification command: `npm run test:unit`,
  `npm run typecheck --prefix apps/api`, plus the Slice 1
  smoke list above.
- Risks: changing the event shape can break the conformance
  smoke; preserve the field names the existing smoke asserts.
- Do not claim yet: compliance-grade audit, retention, or
  export.

### Slice 2 — Restart-persistent audit

- Intended files touched: confirm existing storage behavior
  before touching anything; if audit already rehydrates via
  storage, the slice is mostly the new smoke; if not, add the
  smallest storage-backed audit path consistent with the
  existing architecture.
- Preferred script: `scripts/smoke-audit-durability.mjs`, root
  script `smoke:audit-durability`.
- Tests: restart preserves would-block event with the same ID
  and timestamp; list order remains monotonic.
- OpenAPI impact: none.
- Verification command: `npm run smoke:audit-durability` plus
  the Slice 2 smoke list above.
- Risks: silently mixing memory and persistent modes produces
  misleading evidence; the smoke must explicitly set
  `WARD_STORAGE=sqlite` and bind `WARD_SQLITE_PATH` to a known
  directory.
- Do not claim yet: production durability, cross-node
  durability, retention, export.

### Slice 3 — Per-tenant mode override

- Intended files touched: `apps/api/src/types.ts`,
  `apps/api/src/config.ts`, `apps/api/src/server.ts` (or a new
  small helper), `apps/api/src/audit.ts` if a helper fits,
  `openapi/ward.v0.yaml` if the control surface widens,
  `scripts/smoke-tenant-mode-override.mjs`,
  `scripts/verify-release.sh`, `docs/CLAIMS_AND_EVIDENCE.md`,
  `docs/BUILD_STATUS.md`.
- Behavior: global default via `WARD_MODE` plus a per-tenant
  field (`inherit`/`observe`/`enforce`); effective mode
  resolved per request; change events audited.
- Tests: effective-mode helper, override observe under global
  enforce, override enforce under global observe, inherit
  behavior, override-change audit event.
- OpenAPI impact: extend the health field if useful; document
  the override surface if exposed via HTTP.
- Verification command: `smoke:tenant-mode-override` plus the
  Slice 3 list above.
- Risks: override must not weaken missing-tenant 400, control
  auth 401, or proxy fail-closed 503.
- Do not claim yet: per-tenant control over production routing
  or "off" mode.

### Slice 4 — State transition contract

- Intended files touched: `apps/api/src/tenantState.ts`,
  `apps/api/src/server.ts` if shape changes, unit tests, smoke
  if route behavior changes, possibly a small transition helper.
- Implementation sketch: enumerate allowed transitions; reject
  invalid ones with a clear WardError; ensure
  previousState/nextState are captured in audit; keep
  approval expiry/reuse behavior intact.
- Tests: allowed transitions (running→constrained,
  constrained→paused with approval if required, paused→running
  with approval if required), invalid rejection,
  previousState/nextState captured, reason captured,
  single-use approval token still cannot be reused.
- OpenAPI impact: minimal if any.
- Verification command: Slice 4 list above.
- Risks: tightening transitions can break the demo smoke; the
  demo smoke must remain green.
- Do not claim yet: enterprise workflow engine.

### Slice 5 — Incident receipt export

- Intended files touched: `scripts/export-incident-receipt.mjs`,
  root script `incident:export` or `export:incident`,
  `scripts/smoke-incident-receipt.mjs`,
  `scripts/smoke-incident-receipt` root script, possibly a
  small `apps/api/src/receipt.ts` if the script needs shared
  logic.
- Receipt shape (Markdown): generated timestamp, tenant ID,
  audit event count, relevant event IDs, state transition
  summary, would-block / block summary, approval summary if
  present, limitations section.
- Tests: smoke creates a small scenario with a state change and
  a would-block evidence event, exports the receipt, and
  verifies it contains the required fields.
- OpenAPI impact: none.
- Verification command: Slice 5 list above.
- Risks: wording stronger than the evidence; the export must
  not introduce "compliance", "tamper-proof", or
  "forensic-grade" wording.
- Do not claim yet: legal, compliance, or tamper-proof
  receipt.

### Slice 6 — Failure-behavior verification

- Intended files touched: `scripts/smoke-rc3-failure-behavior.mjs`,

5. `Define Ward tenant state transition contract`
6. `Add incident receipt export for Ward audit evidence`
7. `Verify Ward RC3 failure behavior`

committing is authorized, commit only after the slice's
verification passes.
