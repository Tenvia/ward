## Quickstart Path Added or Updated

`docs/EVALUATOR_QUICKSTART.md` already provides the copy/paste path:

1. create explicit `.env`
2. set `WARD_IMAGE=ghcr.io/tenvia/ward-api:v0.1.0-rc4`
3. set `WARD_REQUIRE_CONTROL_TOKEN=true`
4. generate `WARD_CONTROL_TOKEN`
5. start `docker compose --env-file .env -f docker-compose.pull.yml up -d`
6. verify `/health`, `/openapi.yaml`, and `/`
7. verify unauthenticated control mutation returns `401`
8. constrain Globex with the token
9. confirm Globex returns `429` and Acme returns `200`
10. inspect audit
11. stop with `docker compose --env-file .env -f docker-compose.pull.yml down`

This session updated `docker-compose.pull.yml` and related docs so the default pull image aligns with the published RC4 baseline.

## SQLite and Recovery Notes Added or Updated

`docs/SQLITE_DEPLOYMENT.md` already documents the SQLite scope and was audited against the storage skill facts:

- SQLite persists tenant state and audit events only.
- Approval tokens and workflow runs remain memory-only.
- SQLite state survives API/container restart when the same file/volume is retained.
- `docker compose down --volumes`, deleting the DB file, or changing `WARD_SQLITE_PATH` loses state.
- Backup/restore is offline filesystem copy only.
- `node:sqlite` is experimental in Node 22 and remains prototype-only.
- SQLite is not HA, not shared multi-replica state, not compliance retention, and not tamper-evident.

No storage code changed.
