# Ward v0.1.0-rc4 — Deployment Readiness plan

## Status

RC4 is **in progress**, not release-complete. Slice 0 created this
planning document. Slice 1 added the environment contract as docs-only
work. No RC4 runtime code, script, compose, tag, image publish, README
badge update, or completed release note has been produced.

RC3 is the baseline. `v0.1.0-rc3` is the latest published prototype
baseline recorded in `docs/PUBLISH_READINESS.md` and
`docs/releases/v0.1.0-rc3.md`: versioned local audit events, SQLite
audit restart recovery, per-tenant observe/enforce mode overrides, the
explicit tenant state transition contract, local incident receipt export,
and RC3 failure-behavior coverage are the starting point for RC4.

RC3 boundaries remain in force throughout RC4 unless a later RC4 slice
removes one with matching evidence in the claims ledger and build
status: Ward is not production-ready, not design-partner-ready by
default, not hosted, not compliance-ready, not HA, and not
pass-through-streaming-ready against real upstreams.

## Product question

Can a technical evaluator pull Ward, configure it safely, run it with the
right environment variables, verify it, understand failure behavior, and
operate the prototype without reading the whole repo?

## RC4 promise

Ward RC4 aims to make the verified prototype deployable and operable by a
technical evaluator using documented local/container workflows, explicit
environment configuration, release verification, and operator runbooks.

This is a local/container prototype promise. It is not a v1.0 production
promise, not hosted Ward, not enterprise auth, not Postgres HA, and not a
compliance posture.

## What RC4 should prove

1. A published Ward image can be pulled and verified from the docs using
   the documented GHCR and Docker Compose path.
2. Environment variables are documented, grouped by purpose, and either
   validated or smoke-tested where validation is small and safe.
3. Control auth setup is explicit and testable, including the difference
   between the local demo default and an auth-enabled evaluator path.
4. SQLite storage mode is documented with clear restart-recovery scope,
   persistence limits, backup/restore expectations, and explicit
   non-HA/non-shared-state caveats.
5. Health, served OpenAPI, and Control Room checks are documented as the
   first operational checks after boot.
6. Release verification includes RC3 gates and any RC4 gates added during
   implementation. The existing `./scripts/verify-release.sh` already
   runs the RC3-specific smokes; an RC4 implementation slice may extend
   that existing verifier, but that is a future script change requiring
   explicit slice approval and evidence.
7. Post-publish GHCR verification is repeatable: pull image, boot image,
   wait for health, confirm served OpenAPI, and run the applicable smoke
   or containment check.
8. Operator runbooks exist for common prototype incidents and stay scoped
   to what the current prototype can actually do.
9. Failure behavior is documented without overstating production
   guarantees: degraded proxy fail-open/closed, SDK fail-open/closed,
   observe-mode would-blocks, upstream errors, missing control auth, and
   hard-down gaps are kept separate.
10. Docs and claims distinguish local/container prototype deployment from
    production deployment everywhere the topic appears.

## What RC4 does not prove

RC4 must explicitly **not** be used as evidence for:

- production readiness
- design-partner readiness unless separately accepted by a later gate
- SOC2 or compliance readiness
- hosted Ward
- Postgres, HA, or multi-replica durability
- enterprise RBAC, OIDC, per-operator identity, or token rotation
- tamper-proof or signed receipts
- pass-through streaming against real upstreams
- full OpenAI compatibility
- Kubernetes production deployment

## Proposed RC4 slices

### Slice 0 — RC4 planning alignment

- Create `docs/releases/v0.1.0-rc4.md`.
- Inspect the current docs, compose files, API config, server health
  response, OpenAPI contract, release verifier, and publish workflow.
- No code changes.
- No script, compose, contract, README badge, tag, publish, or push.

### Slice 1 — Environment contract

- Document every supported environment variable from the current API and
  container paths.
- Separate variables into required, optional, prototype-only, and
  dangerous/dev-only categories.
- Include at least:
  - control auth: `WARD_REQUIRE_CONTROL_TOKEN`, `WARD_CONTROL_TOKEN`
  - storage: `WARD_STORAGE`, `WARD_SQLITE_PATH`
  - upstream: `WARD_UPSTREAM_OPENAI_BASE_URL`, `OPENAI_API_KEY`
  - failure behavior: `WARD_PROXY_FAIL_MODE`, `WARD_MODE`
  - detection/cost knobs: `WARD_LOOP_WINDOW_MS`,
    `WARD_LOOP_REQUEST_THRESHOLD`, `WARD_ESTIMATED_COST_PER_REQUEST`,
    `WARD_APPROVAL_TTL_MS`
  - runner/dev flags: `WARD_ENABLE_DOCKER_RUNNER`,
    `WARD_DOCKER_ALLOWED_IMAGES`, `WARD_DOCKER_RUN_TIMEOUT_MS`,
    `WARD_TEST_FORCE_POLICY_ERROR`
  - process/deployment: `PORT`, `WARD_DEPLOYMENT_MODE`
- Add validation or smoke coverage only if small, safe, and directly tied
  to preventing evaluator misconfiguration.
- Keep control auth as shared-token prototype language; do not imply
  production RBAC.

### Slice 2 — Secure local/container quickstart

- Make the evaluator path clearer:
  - pull the published image
  - run Ward with control auth enabled and a non-demo token
  - hit `/health`
  - fetch `/openapi.yaml`
  - open the Control Room
  - run a tenant containment check
  - stop cleanly
- Keep the quickstart framed as local/container prototype evaluation, not
  production deployment.
- Call out when default compose paths differ:
  - `docker-compose.pull.yml` enables control auth by default with a demo
    token unless overridden.
  - root `docker-compose.yml` currently leaves control auth disabled
    unless env vars are set.
- Avoid broad deployment hardening that belongs to production readiness.

### Slice 3 — SQLite deployment/recovery note

- Document what SQLite mode is good for: local/container prototype
  persistence of tenant state and audit across API restarts.
- Document what it is not good for: HA, multi-replica shared state,
  compliance retention, cross-node durability, or production audit
  storage.
- Document restart recovery using the evidence already proven by RC3.
- Document backup/restore expectations if the current volume/file layout
  makes a safe procedure possible.
- Document the future Postgres/shared-state gap without making it RC4
  scope.
- Note that `node:sqlite` remains experimental in Node 22.

### Slice 4 — Operator runbooks

Add short prototype-scoped runbooks for:

- tenant paused accidentally
- constrained tenant would-blocks in observe mode
- control auth missing or misconfigured
- upstream failure
- SQLite file or volume missing
- incident receipt export
- release verifier failure

Each runbook should include symptoms, first checks, safe recovery steps,
verification commands or curls, and explicit prototype limits. Runbooks
must not introduce production SLA, compliance, HA, or enterprise-auth
language.

### Slice 5 — Post-publish verification hardening

- Make post-publish GHCR verification repeatable for each published RC.
- Prefer extending the existing release verification path where practical;
  `./scripts/verify-release.sh` already includes RC3 gates, so RC4 gates
  should be added deliberately rather than creating a competing verifier.
- Consider one small script only if it remains boring and safe:
  - pull image
  - boot image
  - wait for `/health`
  - confirm `/openapi.yaml`
  - run the applicable smoke/openapi/demo checks
- This is a future implementation slice. Modifying
  `scripts/verify-release.sh` or adding a post-publish script is **not**
  part of Slice 0 and should be explicitly accepted before the script is
  changed.
- Do not overbuild registry automation or deployment orchestration.

### Slice 6 — Deployment docs/claims sync

- Update the claims ledger and docs with only the RC4 evidence actually
  produced.
- Soften any wording that implies production deployment,
  design-partner-readiness, compliance, HA, hosted Ward, or enterprise
  auth.
- Reconcile at least the deployment model, publish readiness notes, build
  status, README links/text where needed, and this RC4 document.
- Keep every claim mapped to the status vocabulary in
  `docs/CLAIMS_AND_EVIDENCE.md`.

### Slice 7 — Final RC4 verification

- Run the full final RC4 battery.
- Run `./scripts/verify-release.sh` and ensure it includes the intended
  RC3/RC4 gates before human review/tagging.
- Produce the human review/tagging report with exact commands and
  outcomes.
- Confirm docs distinguish local/container prototype deployment from
  production deployment.
- Confirm `README.md` and `docs/CLAIMS_AND_EVIDENCE.md` do not overclaim.
- Confirm the worktree is clean except for approved changes before any
  tag request.
- Confirm private agent files remain untracked.
- No tag, push, publish, or README badge change without explicit owner
  approval.

## Evidence expectation per implemented slice

Every implemented RC4 slice must leave behind evidence before it is
called complete:

- exact files changed
- commands run
- expected output or observed output
- what claim the evidence supports
- what remains unverified

Screenshots are optional and only useful when the UI is the evidence.
Claims ledger sync is a gate for release readiness, not cosmetic
cleanup.

## Acceptance criteria

RC4 can be considered ready for human review/tagging only if all of the
following are true:

- all planned RC4 slices are complete
- `npm run validate:openapi` passes
- `./scripts/verify-release.sh` passes
- RC3 smokes still pass
- any new RC4 smokes pass
- docs clearly distinguish local/container prototype deployment from
  production deployment
- README and the claims ledger do not overclaim
- `git status --short` is clean
- private agent files remain untracked
- any change to `scripts/verify-release.sh` or a new post-publish
  verifier is documented as an RC4 implementation change with its own
  evidence, not hidden inside planning

## Verification battery proposal

Proposed final RC4 battery, preserving the RC3 floor:

```bash
npm run test:unit
npm run typecheck --prefix apps/api
npm run validate:openapi
npm run smoke:observe-only
npm run smoke:pass-through:mock
npm run smoke:streaming:mock
npm run smoke:audit-durability
npm run smoke:tenant-mode-override
npm run smoke:incident-receipt
npm run smoke:rc3-failure-behavior
npm run smoke:openapi
npm run smoke:reliability
npm run smoke:demo
npm run smoke:sdk
npm run bench:latency
./scripts/verify-release.sh
```

Possible RC4-specific checks, planned names only unless a later slice
implements them:

```bash
npm run smoke:env-contract          # planned; only if Slice 1 adds validation/smoke coverage
npm run smoke:deployment-quickstart # planned; only if Slice 2 adds a small evaluator-path smoke
npm run smoke:post-publish:ghcr     # planned; only if Slice 5 adds repeatable GHCR verification
```

If the RC4-specific checks are added, the final verifier should either
run them directly or state why a separate post-publish command is the
required gate. The preferred path is one release verification story, not
parallel scripts with unclear authority.

## Claim language

### Allowed RC4 target claim

Ward can be pulled, configured, smoke-tested, and operated as a
documented local/container prototype with explicit environment,
verification, and runbook guidance.

### Forbidden RC4 claim language

Do not use RC4 to claim any of the following:

- production-ready
- production-grade deployment
- design-partner-ready
- SOC2-ready
- compliant
- hosted
- HA
- multi-replica
- enterprise auth
- tamper-proof
- complete OpenAI-compatible gateway
- real upstream pass-through streaming

## Open risks

- Current control auth is a shared-token prototype only.
- The root local container stack boots with control auth disabled unless
  `WARD_REQUIRE_CONTROL_TOKEN=true` and `WARD_CONTROL_TOKEN` are set.
- The pull-based compose file enables control auth by default but uses a
  demo token unless the evaluator overrides it.
- SQLite is local prototype state, not HA or shared state.
- `node:sqlite` remains experimental in Node 22.
- Approval tokens and workflow runs remain in memory even when SQLite is
  enabled.
- Post-publish verification may still require manual orchestration until
  Slice 5 hardens it.
- Smoke scripts rely on local ports and process cleanup.
- Extending `./scripts/verify-release.sh` with RC4 gates is likely
  necessary for a single release authority, but it is a future script
  change and must be approved in its own implementation slice.
- The GHCR workflow emitted a non-blocking Node.js 20 deprecation
  annotation during RC3 publish verification.
- No Kubernetes production deployment exists.
- No Postgres backend exists.
- No hosted Ward exists.
- Pass-through streaming against a real upstream remains unproven.
- Full OpenAI compatibility remains unproven.
