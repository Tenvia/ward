## Product question

Can a technical evaluator understand what Ward is, why it matters, how to try it, how to verify it, and what its current limits are within the first 30 minutes?

## RC5 promise

Ward presents a clear, honest, technically grounded evaluator experience for the local/container open-source prototype, with a coherent README path, demo path, verification path, common-error guidance, and claim boundaries.

This is an evaluator-experience promise. It is not a production-readiness promise, not hosted Ward, not HA, not enterprise auth, not SOC2/compliance posture, not a complete OpenAI-compatible gateway, and not proof of real upstream pass-through streaming.

## What RC5 should prove

1. A first-time technical evaluator can identify the right starting path from the README without reading the whole repo.
2. The top-level README explains Ward's problem, mechanism, prototype scope, and safe evaluator path before deeper docs.
3. The demo path is easy to find and does not conflict with the secure evaluator quickstart.
4. Verification remains visible: evaluators see health, OpenAPI, Control Room, containment, and cleanup checks early.
5. Common-error guidance is discoverable without duplicating every runbook in the README.
6. Public wording stays aligned with `docs/CLAIMS_AND_EVIDENCE.md` and RC4's prototype boundaries.
7. Any screenshots or visual proof are either real captured evidence or explicitly deferred; no fake image assets are added.

## What RC5 does not prove

RC5 must explicitly **not** be used as evidence for:

- production readiness
- production-grade deployment
- hosted Ward
- HA or multi-replica operation
- enterprise auth, RBAC, OIDC, per-operator identity, or token rotation
- SOC2 or compliance readiness
- tamper-proof or signed receipts
- a complete OpenAI-compatible gateway
- pass-through streaming against a real upstream unless separately verified
- full OpenAI compatibility unless separately verified
- design-partner readiness unless a later slice explicitly supports it with project evidence

## Target evaluator

The primary RC5 evaluator is a technical SaaS operator, founder, staff engineer, or platform engineer who:

- understands Docker, curl, HTTP APIs, and environment variables
- is evaluating tenant-scoped AI containment for an existing multi-tenant SaaS
- wants to see a concrete Acme/Globex demo quickly
- needs honest failure behavior and prototype limits before trusting the project
- should not need to read every document in `docs/` before the first successful run

## First 30-minute evaluator journey

### Minute 0-3 — Understand the wedge

Start at `README.md` and answer:

- Ward sits between a SaaS and LLM/tool APIs as a tenant-aware egress proxy and control plane.
- The demo promise is: one tenant can be constrained or paused while other tenants continue.
- Ward is a local/container prototype, not production infrastructure.

### Minute 3-10 — Boot the safe evaluator path

Use `README.md#first-60-seconds` or `docs/EVALUATOR_QUICKSTART.md`:

- create `.env`
- set `WARD_IMAGE=ghcr.io/tenvia/ward-api:v0.1.0-rc4`
- set `WARD_REQUIRE_CONTROL_TOKEN=true`
- generate `WARD_CONTROL_TOKEN`
- start `docker compose --env-file .env -f docker-compose.pull.yml up -d`

### Minute 10-15 — Verify the surface

Confirm:

- `GET /health`
- `GET /openapi.yaml`
- bundled Control Room at `/`
- missing-token control mutation returns `401`

### Minute 15-25 — Prove containment

Run the Acme/Globex sequence:

- constrain `tenant_globex` with the bearer token
- confirm Globex receives `429`
- confirm Acme still receives `200`
- inspect `/ward/audit`

### Minute 25-30 — Understand limits and next docs

Read:

- `README.md#what-ward-is-not`
- `docs/ENVIRONMENT.md` for supported env vars and demo-token warnings
- `docs/OPERATOR_RUNBOOK.md` for first failure checks
- `docs/CLAIMS_AND_EVIDENCE.md` before repeating claims

- `docs/releases/v0.1.0-rc5.md` exists and defines the evaluator-experience scope.
- README clearly routes first-time technical evaluators to the safe local/container path.
- Demo-token, control-auth, SQLite, and prototype-limit language remain honest.
- Links in the README's primary evaluator path point to real files or anchors.
- `docs/CLAIMS_AND_EVIDENCE.md` remains the claim authority.
- No runtime behavior changed unless a later slice explicitly chooses and verifies it.
- `npm run validate:openapi` passes, or failure is recorded clearly.
- Any broader verification that is skipped is named with a reason.
- `git status --short` contains only intentional RC5 changes before commit, then is clean after any commit.
- Private agent files remain untracked.
- No tag, push, publish, deploy, or GitHub Release action occurred during RC5 work without explicit owner instruction.
